项目名称：WebRTC PCDN 级联直播系统 - 四期网络调度与去中心化需求
版本目标：从中心化信令调度走向真正的P2P自治网络

一、核心设计理念：最小中心化
1.1 背景与痛点
前三期的实现中，信令服务器承担了越来越多的职责：房间管理、拓扑维护、父节点选择、备用节点分配、故障转移调度、统计汇总等。这违背了P2P系统的本质——信令服务器变成了"上帝节点"，一旦信令服务器宕机，整个网络瘫痪。

真正的P2P系统中，信令服务器应该只做"引导"（Bootstrap），不做"管理"。

1.2 设计原则

信令服务器的职责收缩为：
  - 房间发现：新节点通过信令获取房间内若干初始节点（Bootstrap Nodes）的信息
  - SDP/ICE交换：仅在两个节点首次建立连接时中转信令（一旦DataChannel建立，后续信令走DataChannel）
  - 主播注册：主播创建房间时注册，提供房间元信息

信令服务器不再存储：
  - 完整拓扑结构（谁是谁的父节点、子节点）
  - 节点的就绪状态、统计信息
  - 备用节点分配关系

状态分散管理：
  - 每个节点自己维护：我的父节点是谁、我的子节点有哪些、我的备用连接列表
  - 拓扑信息通过DataChannel在节点间传播（Gossip协议）
  - 新节点加入后，通过DataChannel从邻居节点获取最新的全局拓扑视图

1.3 信令服务器精简后的接口

  createRoom(roomId, publisherInfo) → 注册房间，返回房间token
  getRoomInfo(roomId) → 返回房间元信息 + 若干Bootstrap节点列表（最多5-8个活跃节点）
  relaySignal(targetId, sdp/candidate) → 仅用于首次连接建立时的SDP/ICE交换
  heartbeat(roomId, peerId) → 节点定期心跳，信令服务器仅维护"活跃节点列表"（不维护拓扑关系）

  信令服务器存储的状态极简：
  {
    roomId: "room1",
    publisher: { peerId, nickname, createdAt },
    activeNodes: [ { peerId, nickname, lastSeen } ],  // 仅活跃节点列表，无拓扑关系
    password: "...",
    title: "..."
  }

二、链路感知与本地网络发现
2.1 背景与痛点
当前系统对所有节点一视同仁，不区分同一局域网内的节点和跨广域网的节点。实际上，同一局域网（同一WiFi、同一公司网络）内的节点之间P2P传输速度极快、延迟极低，是最理想的级联分发路径。

2.2 功能需求
2.2.1 本地网络探测

节点加入网络后，通过ICE Candidate信息判断与其他节点的网络关系：
  - 同一局域网（host candidate匹配同一子网）：标记为"本地节点"
  - 同一NAT后（srflx candidate的公网IP相同）：标记为"同NAT节点"
  - 不同网络（公网IP不同）：标记为"远程节点"

探测方法：
  - 方法一：在ICE连接建立后，检查选中的candidate pair类型
    - host-host：同一局域网
    - srflx-srflx且公网IP相同：同一NAT
    - relay或不同公网IP：远程
  - 方法二：通过DataChannel发送探测包，测量RTT
    - RTT < 5ms：极大概率同一局域网
    - RTT < 30ms：可能同一城域网
    - RTT > 50ms：跨区域

2.2.2 链路分类标签

每个连接标记链路类型：
  - LOCAL：同一局域网，RTT < 5ms
  - NEARBY：同一NAT或城域网，RTT < 30ms
  - REMOTE：跨区域，RTT > 30ms
  - RELAY：通过TURN中继

2.2.3 基于链路感知的父节点选择

优先选择LOCAL节点作为父节点（带宽大、延迟低、不消耗公网带宽）
其次选择NEARBY节点
最后选择REMOTE节点
RELAY节点作为最后手段

选择算法伪代码：
  function selectBestParent(candidates) {
    // 按链路类型分组
    const local = candidates.filter(c => c.linkType === 'LOCAL');
    const nearby = candidates.filter(c => c.linkType === 'NEARBY');
    const remote = candidates.filter(c => c.linkType === 'REMOTE');

    // 每组内按质量评分排序（RTT、丢包率、可用扇出数）
    // 优先从local中选择，local无可用则nearby，依次降级
    for (const group of [local, nearby, remote]) {
      const available = group.filter(c => c.fanoutAvailable > 0);
      if (available.length > 0) {
        return available.sort((a, b) => a.rtt - b.rtt)[0];
      }
    }
    return null;
  }

2.2.4 拓扑可视化增强

拓扑图中用不同颜色/线型标识链路类型：
  - LOCAL连接：绿色粗实线
  - NEARBY连接：蓝色实线
  - REMOTE连接：灰色实线
  - RELAY连接：红色虚线

三、多备用连接与智能故障转移
3.1 背景与痛点
三期实现了主+备双连接，但在大规模网络中，单个备用连接可能不够。特别是当网络分区发生时，需要跨分区的备用连接来保证连通性。

3.2 功能需求
3.2.1 动态备用连接数量

根据网络规模和节点位置，动态决定备用连接数量：
  - 小型网络（<10节点）：1个备用
  - 中型网络（10-50节点）：2个备用
  - 大型网络（>50节点）：2-3个备用

备用连接的选择策略（基于链路感知）：
  - 备用1：优先选择与主连接不同链路类型的节点（多样性）
    例如主连接是LOCAL节点，备用选择REMOTE节点，防止局域网整体断开
  - 备用2：选择不同子树的节点（拓扑多样性）
    防止某个子树整体断开导致所有备用失效
  - 备用3（如有）：选择距离根节点最近的可用节点（低延迟保障）

3.2.2 备用连接维护策略

备用连接建立后的维护方式：
  - 轻量模式：仅维持DataChannel连接，不传输媒体流
    优点：几乎不消耗带宽
    缺点：切换时需要重新协商媒体，有短暂中断
  - 热备模式：备用连接也接收媒体流（可降低质量）
    优点：切换几乎无感知
    缺点：消耗额外带宽
  - 混合模式（推荐）：备用连接接收音频流+低频关键帧
    折中方案，切换时音频无缝，视频短暂模糊后恢复

3.2.3 故障转移决策

故障检测信号：
  - ICE连接状态变为disconnected/failed
  - DataChannel关闭
  - 媒体流停滞（bytesReceived不增长超过阈值）
  - RTT突然飙升（超过正常值3倍以上）

故障转移流程：
  1. 检测到主连接异常
  2. 立即探测所有备用连接的健康状态（通过DataChannel ping）
  3. 选择最健康的备用连接提升为主连接
  4. 如果备用连接也不可用，通过DataChannel向邻居节点请求推荐新的父节点
  5. 如果DataChannel也不可用（完全断网），回退到信令服务器请求Bootstrap节点
  6. 建立新的主连接后，异步建立新的备用连接

3.2.4 故障转移的通知与传播

当节点A的父节点断开，A切换到备用节点B后：
  - A通过DataChannel通知B："我现在是你的子节点"
  - B更新自己的子节点列表
  - A通过DataChannel向自己的子节点广播拓扑变更
  - 拓扑变更通过Gossip协议在网络中传播

四、拓扑状态的去中心化管理
4.1 背景与痛点
当前拓扑状态完全由信令服务器维护和广播。这不仅增加了信令服务器的负担，也意味着信令服务器是单点故障。

4.2 功能需求
4.2.1 节点本地状态

每个节点维护自己的状态：
  {
    peerId: "xxx",
    nickname: "观众189",
    role: "viewer",           // publisher / viewer
    parentId: "yyy",          // 我的主父节点
    backupParentIds: ["zzz", "www"],  // 我的备用父节点列表
    children: ["aaa", "bbb"], // 我的子节点列表
    linkTypes: {              // 与各节点的链路类型
      "yyy": "LOCAL",
      "zzz": "REMOTE",
      "aaa": "NEARBY"
    },
    stats: {                  // 我的连接质量
      rtt: { "yyy": 3, "zzz": 120 },
      packetLoss: { "yyy": 0, "zzz": 0.5 }
    },
    isReady: true,            // 是否可以转发
    joinedAt: 1234567890,
    fanoutMax: 2,             // 最大扇出数
    fanoutAvailable: 0        // 剩余可用扇出
  }

4.2.2 拓扑信息的Gossip传播

每个节点定期（如每5秒）通过DataChannel向邻居节点发送自己的状态摘要
邻居节点收到后，更新本地的全局拓扑视图，并继续向自己的邻居传播
使用版本号或时间戳防止过期信息覆盖新信息

Gossip消息格式：
  {
    type: "topology-gossip",
    version: 42,              // 递增版本号
    timestamp: 1234567890,
    nodes: [
      { peerId, parentId, children, isReady, linkType, rtt, fanoutAvailable }
    ]
  }

传播策略：
  - 每个节点向所有直接连接的节点（父+子+备用）发送
  - 收到新版本的拓扑信息后，合并到本地视图，然后转发
  - 已经见过的版本号不再转发（防止循环）
  - 拓扑信息带TTL（如最多传播5跳），防止大规模网络中的广播风暴

4.2.3 新节点加入流程（去中心化版）

  1. 新节点连接信令服务器，获取房间的Bootstrap节点列表（5-8个活跃节点）
  2. 新节点通过信令服务器与Bootstrap节点交换SDP/ICE，建立DataChannel
  3. 通过DataChannel从Bootstrap节点获取最新的全局拓扑视图
  4. 根据拓扑视图，选择最优的父节点（基于链路感知+负载均衡）
  5. 通过DataChannel（经由已连接的节点中转信令）与目标父节点建立WebRTC连接
  6. 连接建立后，新节点开始接收媒体流
  7. 新节点通过Gossip协议向网络广播自己的加入

4.2.4 节点离开的处理

正常离开：
  - 节点通过DataChannel向父节点和子节点发送"离开"消息
  - 子节点立即切换到备用连接
  - 拓扑变更通过Gossip传播

异常离开（崩溃/断网）：
  - 子节点通过ICE状态检测到父节点断开
  - 子节点切换到备用连接
  - 备用连接的父节点通过Gossip广播拓扑变更
  - 其他节点更新本地拓扑视图，移除离线节点

4.2.5 DataChannel信令中转

当两个不直接相连的节点需要建立WebRTC连接时：
  - 通过已有的DataChannel链路中转SDP/ICE信令
  - 路由策略：沿着拓扑树找到两个节点的最短路径
  - 中转消息格式：
    {
      type: "relay-signal",
      fromId: "新节点",
      targetId: "目标父节点",
      hops: ["中间节点1", "中间节点2"],
      data: { type: "offer", sdp: "..." }
    }
  - 每个中间节点收到后，检查targetId是否是自己，不是则沿路径转发

五、自适应拓扑调度策略
5.1 背景与痛点
当某个父节点的发送质量下降时（如带宽不足、CPU过载），需要一套平衡策略来决定：是让其他父节点分摊负载，还是让子节点主动迁移。

5.2 功能需求
5.2.1 质量下降的检测与分类

轻度下降（RTT上升50%或丢包率1-3%）：
  - 可能是暂时性波动，观察等待
  - 不触发任何调度动作

中度下降（RTT上升100%或丢包率3-10%）：
  - 通知子节点准备备用连接
  - 父节点主动降低发送码率（如从720p降到480p）
  - 如果父节点有多个子节点，考虑让部分子节点迁移

重度下降（RTT上升200%以上或丢包率>10%）：
  - 子节点立即切换到备用连接
  - 父节点标记为"降级节点"，不再接受新的子节点
  - 通过Gossip广播该节点的降级状态

5.2.2 负载均衡策略

当父节点A负载过高时，调度策略：

策略一：子节点迁移（推荐用于中度下降）
  - A的部分子节点迁移到其他有空余扇出的节点
  - 迁移优先级：选择与目标节点链路质量最好的子节点
  - 迁移过程：子节点先建立到新父节点的连接，确认流畅后再断开与A的连接

策略二：负载分摊（推荐用于轻度下降）
  - A降低发送码率，子节点同时从备用连接补充数据
  - 类似于多源下载，从多个上游拼接完整的流
  - 实现复杂度高，四期可作为实验性功能

策略三：主动降级（推荐用于重度下降）
  - A主动通知所有子节点："我即将降级，请切换"
  - 子节点收到通知后，有序地切换到备用连接
  - 比被动检测更快，减少中断时间

5.2.3 全局拓扑优化（周期性）

每个节点定期（如每30秒）评估自己的连接质量
如果发现更优的父节点（基于链路感知评分），主动发起迁移
迁移条件：新父节点的评分比当前父节点高出20%以上（避免频繁切换）

优化目标：
  - 最小化全局平均延迟
  - 最大化LOCAL连接比例
  - 均衡各节点的扇出负载
  - 最小化树的深度（减少级联延迟）

六、四期验收标准

去中心化验证：
  - 信令服务器仅存储房间元信息和活跃节点列表，不存储拓扑关系
  - 新节点通过Bootstrap节点和DataChannel完成加入，信令服务器仅参与首次SDP交换
  - 信令服务器短暂断开（如10秒）后恢复，已在网络中的节点不受影响

链路感知验证：
  - 同一局域网内的节点能被正确识别为LOCAL
  - 父节点选择优先使用LOCAL节点
  - 拓扑图中能清晰区分不同链路类型

多备用连接验证：
  - 中型网络中每个节点维持2个备用连接
  - 主连接断开后，500ms内切换到备用连接，视频播放无明显中断
  - 备用连接来自不同链路类型或不同子树

拓扑Gossip验证：
  - 新节点加入后，10秒内所有节点的本地拓扑视图更新
  - 节点离开后，15秒内所有节点感知到拓扑变更
  - Gossip消息不会产生广播风暴（通过版本号去重和TTL限制）

自适应调度验证：
  - 模拟父节点带宽下降，子节点能在5秒内完成迁移
  - 全局拓扑优化能自动将LOCAL连接比例提升到可能的最大值
  - 频繁切换保护：同一节点30秒内不会发生两次父节点切换

八、技术风险与注意事项

8.1 Gossip协议的收敛速度
  - 在大规模网络中，Gossip传播可能需要多轮才能覆盖所有节点
  - 需要设计合理的传播频率和TTL，平衡收敛速度和带宽消耗
  - 建议：小网络（<20节点）全量广播，大网络使用增量更新

8.2 DataChannel信令中转的可靠性
  - DataChannel中转信令依赖中间节点在线
  - 如果中间节点断开，信令中转失败
  - 回退策略：DataChannel中转失败时，回退到信令服务器中转

8.3 拓扑一致性
  - 去中心化管理意味着各节点的拓扑视图可能短暂不一致
  - 这是可接受的（最终一致性），但需要处理冲突
  - 例如：两个节点同时选择同一个父节点，导致扇出超限
  - 解决方案：父节点收到超限的连接请求时，拒绝后来者，后来者重新选择

8.4 浏览器资源限制
  - 多备用连接会增加PeerConnection数量，注意浏览器的连接数限制
  - Chrome通常支持数十个PeerConnection，但每个都消耗内存和CPU
  - 建议：总连接数（主+备+子）不超过8-10个

8.5 向后兼容
  - 四期的去中心化改造是架构级变更
  - 建议采用渐进式迁移：先实现DataChannel信令中转和Gossip，再逐步收缩信令服务器职责
  - 保留信令服务器作为回退通道，确保在去中心化机制失效时系统仍可用
